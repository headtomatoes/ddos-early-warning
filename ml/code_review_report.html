BLOCKER
No sklearn Pipeline built — transforms not embedded for ONNX
The critique's core architectural recommendation was: "Rebuild the pipeline so that log1p and the scaler are embedded inside the ONNX graph using sklearn.pipeline.Pipeline → skl2onnx.convert_sklearn." This was not implemented. The script trains a bare XGBClassifier on raw unscaled data and saves it with .save_model() (XGBoost JSON format). When Week 6 exports this to ONNX, the resulting model will have no transform layer. The Go controller must then manually apply log1p before calling the model — which is exactly the silent inference bug the critique flagged. Training on raw data is fine for tree models, but the ONNX export path requires a Pipeline wrapper.
Required implementation for Week 6 ONNX export
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import FunctionTransformer, MinMaxScaler

pipeline = Pipeline([
    ("log1p",  FunctionTransformer(np.log1p, validate=True)),
    ("scaler", MinMaxScaler()),
    ("model",  xgb.XGBClassifier(**best_cfg))
])
pipeline.fit(X_train, y_train, model__sample_weight=current_weights)
# Export entire pipeline — Go receives raw values, ONNX handles transforms internally
from skl2onnx import convert_sklearn
from skl2onnx.common.data_types import FloatTensorType
model_onnx = convert_sklearn(pipeline, initial_types=[('float_input', FloatTensorType([None, 8]))])
with open("xgboost_final.onnx","wb") as f: f.write(model_onnx.SerializeToString())
MEMORY
weights_train.copy() not explicitly released between iterations
weights_train (49M rows × 8 bytes ≈ 392MB) is computed outside the loop. Inside the loop, current_weights = weights_train.copy() allocates another 392MB per iteration. del model + gc.collect() frees the model but not the weight copy. Peak usage is ~784MB across two weight arrays simultaneously. For a 32GB machine this is acceptable, but add del current_weights after each model.fit() call to be precise.

Blocker — not cleared for Dr. Le Hai Duong review yet
The pipeline cannot receive a green light in its current state. The missing sklearn Pipeline wrapper in w5_train_final.py means that the Week 6 ONNX export will produce a model with no transform layer. The Go eBPF controller will then receive raw packet counter values and pass them directly to a model trained on raw values — which is consistent, so inference will not immediately crash. However, it violates the core architectural requirement that transforms are embedded in the ONNX graph, making the system fragile to any future preprocessing change and impossible to validate end-to-end in Python before deployment.
Warning — UDP-LAG still in dataset as a separate class
'UDP-lag' (366,461 rows) becomes 'UDP-LAG' after .str.upper() and is not caught by the replace list. It will remain as a standalone class in the parquet files and in the trained model. This is not catastrophic — UDP-lag has enough samples for the model to learn boundaries — but it contradicts the merge plan and will confuse the thesis narrative about rare class elimination.
