criticalw5_optimize trains a model that is never saved — the stage is orphaned
w5_optimize.py trains xgb_optimized with max_depth=6 and produces the confusion matrix, but it is never persisted to disk. Then w5_tune.py trains a completely independent model from scratch (max_depth=5) and saves it as xgboost_final.json. The optimize → tune relationship you intended is simply two separate training runs. All compute spent in w5_optimize that isn't the confusion matrix is wasted. The pipeline is not sequential — it is parallel and disconnected.
Fix: Either save the optimized model with xgb_optimized.save_model('xgb_optimized.json') and load it in w5_tune for further fine-tuning, or collapse the two stages into one script with a clear progression: train baseline → evaluate → adjust hyperparameters → retrain → save final.
criticalLabelEncoder is re-fit from scratch in every script — silent label-mapping risk
Each of w4_train, w5_optimize, w5_tune, and w5_threshold calls le.fit_transform(y_train) or le.fit_transform(y_val) independently. Since sklearn sorts class names alphabetically the mapping is currently consistent, but this breaks silently if any class is absent from a data split. For example: if the stratified split puts zero WEBDDOS rows (only 439 total) in one split, that script's LE maps classes differently and all numeric labels become wrong with no error thrown.
Fix: Fit LabelEncoder once in w3_preprocess.py, save with joblib.dump(le, 'label_encoder.pkl'), and load it in every downstream script. Never re-fit.
warningPer-class imputation uses apply(lambda row:…) — catastrophically slow at 70M rows
The imputation code in w3_preprocess.py iterates every row with a Python-level lambda on a 70M-row DataFrame. This is O(n) in pure Python and will take tens of minutes to hours. It also causes the imputation to run three times (train, val, test). The log shows this step completed without timing — that's suspicious; it may have been tested on a subset.
Fix — vectorised, 1000× faster: 
class_medians = X_train.groupby('Label', observed=True)[col].median()
X_train[col] = X_train[col].fillna(X_train['Label'].map(class_medians))
X_val[col]   = X_val[col].fillna(X_val['Label'].map(class_medians))
X_test[col]  = X_test[col].fillna(X_test['Label'].map(class_medians))
warningWeight clipping in w5_tune made BENIGN-F1 worse, not better (0.7950 → 0.7606)
The clip rationale was that extreme weights for rare classes (WEBDDOS: 439 rows → weight ≈ 160,000×) were "destroying tree logic." But the clipping reduced BENIGN weight indirectly, since BENIGN's balanced weight is already moderate. The confusion matrix shows the real problem: PORTMAP (precision=0.02), WEBDDOS (precision=0.02), and UDPLAG (precision=0.01) have near-zero precision — they are massively over-predicted. These classes flood the confusion matrix with false positives that contaminate BENIGN evaluation.
Fix: Instead of clipping all weights, drop or merge UDPLAG (1,873 rows) and WEBDDOS (439 rows) into an OTHER_ATTACK group before training. These counts are too small for a tree model to learn meaningful boundaries from.
criticalNo threshold achieves the 0.1% FPR target — the fundamental model hasn't converged
The threshold sweep is unambiguous. The best achievable FPR with this model is 0.322% at threshold=0.01, which misses 665,095 attacks. The 0.1% FPR target cannot be met at any threshold:

Root cause: The 6-feature set (FEATURE_SET_V1) lacks sufficient discriminators to cleanly separate BENIGN from low-rate attacks like PORTMAP and LDAP. The confusion matrix shows BENIGN rows being predicted as DRDOS_MSSQL and SYN — both have overlapping flow byte counts. criticalFeature importance order contradicts EDA — Flow Bytes/s ranked #1 by eta² but #3 by XGBoost gain
Your EDA established Flow Bytes/s as the most discriminative feature (η²=0.6305). XGBoost's gain importance chart ranks it third behind Total Length of Fwd Packets and Total Backward Packets. This divergence has a specific cause: after log1p and MinMax scaling, the raw eta² advantage of Flow Bytes/s compresses. The 3.08% NaN imputation also introduced artificial values. This doesn't invalidate the model, but it means your thesis claim about "data-validated features from EDA" is now partially undermined — XGBoost learned a different ranking than eta² predicted.
Fix for thesis: Explicitly note in model_card.md that post-transform gain importance is the operative metric, not eta². Run a permutation importance test (which is transformation-invariant) to get a clean ranking.
warningRare classes (WEBDDOS=439, UDPLAG=1873) poison precision across the board
From the classification report: PORTMAP precision=0.02, WEBDDOS precision=0.02, UDPLAG precision=0.01. These classes are so rare that the model learns to predict them promiscuously — it's better rewarded for high recall than controlling precision. Every false PORTMAP or WEBDDOS prediction is a false positive for every other class. This cascades into BENIGN being predicted as these classes (visible in the confusion matrix's off-diagonal BENIGN row spread).
Fix: Before training, merge WEBDDOS, UDPLAG, and UDPLag (which are 439, 1873, and clearly duplicated classes) into a single RARE_ATTACK label. This reduces from 19 to 17 classes and removes the precision sinks.
warninglog1p applied outside the ONNX pipeline — inference time transform mismatch risk
w3_preprocess.py applies log1p and MinMaxScaler before saving to parquet. This means the ONNX model receives pre-transformed data. When the Go controller feeds raw eBPF counter values to the ONNX model at inference time, it must replicate the exact same log1p → MinMax sequence in Go before calling the model. This is the #1 source of silent inference errors in production.
Fix (as specified in the revised plan): Rebuild the pipeline for Week 6 so that log1p and the scaler are embedded inside the ONNX graph using sklearn.pipeline.Pipeline → skl2onnx.convert_sklearn. Train on raw (unscaled) parquet. Go passes raw values; the ONNX model handles all transforms internally.

Recommendation 1 — Collapse optimize+tune into a single gated script with automatic rollback
The current design has two separate training runs with no mechanism to compare them before the next step runs. Replace w5_optimize and w5_tune with a single w5_train_final.py that runs training experiments sequentially, compares BENIGN-F1 after each, and only advances if the metric improves. If a configuration regresses (as weight clipping did), it automatically reverts to the previous checkpoint. 
EXPERIMENTS = [
    {"max_depth":4, "n_estimators":100, "weight_clip":None},   # baseline
    {"max_depth":6, "n_estimators":150, "weight_clip":None},   # optimize
    {"max_depth":5, "n_estimators":100, "weight_clip":100.0},  # tune
]
best_benign_f1 = 0
for i, cfg in enumerate(EXPERIMENTS):
    model = train(cfg)
    metrics = evaluate(model, X_val, y_val)
    log_stage(f"EXPERIMENT_{i}", cfg, metrics)
    if metrics["benign_f1"] > best_benign_f1:
        best_benign_f1 = metrics["benign_f1"]
        model.save_model("xgb_best_checkpoint.json")
        log("CHECKPOINT: new best saved")
    else:
        log("ROLLBACK: config regressed, reverting to previous checkpoint")
Recommendation 2 — Feed threshold sweep results directly into a deployment config JSON
Currently w5_threshold.py prints the sweep to stdout and nothing is saved. The Go controller needs a machine-readable threshold to use at inference time. Automate the transition: after the sweep, the script selects the optimal threshold (lowest threshold where FPR ≤ target), writes it to deploy_config.json, and the Go controller reads this file at startup. 
TARGETS = {"max_fpr_pct": 0.10, "max_missed_attack_pct": 5.0}

best = select_threshold(sweep_results, TARGETS)
config = {
    "model_path": "xgboost_final.onnx",
    "threshold":  best["threshold"],
    "achieved_fpr_pct": best["fpr"],
    "achieved_missed_attacks": best["missed"],
    "target_met": best["fpr"] <= TARGETS["max_fpr_pct"],
    "generated_at": datetime.utcnow().isoformat()
}
with open("deploy_config.json","w") as f:
    json.dump(config, f, indent=2)

if not config["target_met"]:
    raise SystemExit("PIPELINE BLOCKED: FPR target not met. Retrain required.")
The raise SystemExit prevents Week 6 ONNX export from running on a model that hasn't passed the gate — enforcing the MVP criteria automatically.

The three most urgent changes to break through the current FPR ceiling, in priority order:

RARE_ATTACK_CLASSES = {'WEBDDOS', 'UDPLAG', 'UDPLAG'}

The class merge eliminates three precision sinks. The flag features add L4-state information that directly separates SYN floods (SYN=N, ACK=0) from legitimate traffic (SYN=1, ACK=0 → SYN=0, ACK=1). These two changes together are the most likely path to BENIGN-F1 ≥ 0.90.
